## 리뷰 결과

**상태**: WARNING  
**요약**: 기능은 전체적으로 잘 설계되었지만, 쉘 스크립트에서 약간의 견고성/유지보수성 이슈가 있습니다.

---

**발견된 이슈**:

- **환경 검증 부족 (`switch-review-storage.sh`, `view-review.sh`)**  
  - 두 스크립트 모두 현재 디렉토리가 Git 저장소인지, `.git/hooks/post-commit`이 존재하는지 등에 대한 명시적인 사전 검증이 없습니다.  
  - `switch-review-storage.sh`는 `.git/hooks/post-commit`에 특정 문자열이 없으면 단순히 “확인할 수 없습니다”만 출력하고 종료하는데, 사용자가 “왜 인식이 안 되는지”를 이해하기 어렵습니다.

- **약간 아쉬운 quoting / 파일 존재 체크 (`switch-review-storage.sh`)**  
  - `if [ -d "$CURRENT_DIR" ] && [ "$(ls -A $CURRENT_DIR 2>/dev/null)" ]; then` 에서 `$CURRENT_DIR`가 따옴표 없이 사용됩니다. 현재 값이 `.git/reviews`/`reviews`라 실질 버그는 아니지만, 관습적으로는 `"$CURRENT_DIR"`로 감싸는 것이 안전합니다.
  - 리뷰 디렉토리 내 파일 검사에 `ls`를 사용하는 패턴은 관습적으로 권장되지는 않습니다 (`find`나 glob + 조건 사용이 더 안전/명료).

- **hook 포맷에 대한 강한 결합 (`switch-review-storage.sh`)**  
  - `grep -q 'REVIEWS_DIR=".git/reviews"' .git/hooks/post-commit` 같은 고정 문자열 매칭에 의존합니다.  
  - Hook 파일이 조금만 다른 형식(공백, 작은따옴표, 변수 정의 위치 변경 등)으로 작성되면 “리뷰 저장 위치를 확인할 수 없습니다”로 떨어집니다. 실제 버그라기보다는 유지보수성/견고성 이슈입니다.

- **에러 메시지의 구체성 부족 (`view-review.sh` 옵션 2)**  
  - 잘못된 커밋 해시를 입력했을 때, `git rev-parse` 실패 시에도 단순히 “해당 커밋에 대한 리뷰가 없습니다”만 출력됩니다.  
  - 실제로는 “유효하지 않은 커밋 해시”와 “리뷰 파일이 없는 정상 커밋”을 구분해 주는 것이 디버깅에 도움이 됩니다.

**개선 제안**:

- **Git 환경/파일 존재 여부를 선제적으로 검증**  
  - `switch-review-storage.sh` 상단에 다음과 같은 체크를 추가하는 것을 권장합니다.
    - `.git` 디렉토리 존재 여부
    - `.git/hooks/post-commit` 파일 존재 여부 (없으면 친절한 안내 메시지)
  - `view-review.sh`에서도 `git rev-parse HEAD` 전에 `git rev-parse --is-inside-work-tree` 등을 통해 “Git 저장소 내에서 실행해야 한다”는 메시지를 명확히 줄 수 있습니다.

- **quoting 및 파일 검사 방식 정리**  
  - `ls -A $CURRENT_DIR` → `ls -A "$CURRENT_DIR"`로 변경해 관례에 맞게 안전성을 높이세요.  
  - `if [ -d "$CURRENT_DIR" ] && [ "$(ls -A "$CURRENT_DIR" 2>/dev/null)" ]; then` 또는 `shopt -s nullglob` + `for` 루프 등으로 `ls` 의존도를 줄이는 것도 좋습니다.

- **hook 설정 탐지 로직을 좀 더 유연하게**  
  - `REVIEWS_DIR` 행을 정규식으로 파싱하여 값만 추출하는 방식으로 바꾸면, 공백이나 따옴표 스타일이 조금 달라도 동작할 수 있습니다.  
    - 예: `grep -E 'REVIEWS_DIR *= *"' .git/hooks/post-commit | head -1 | sed 's/.*REVIEWS_DIR *= *"\(.*\)".*/\1/'` 등.
  - 또는 “현재 모드가 정확히 인식되지 않으면, 양쪽 후보(.git/reviews, reviews)를 검사해서 유추”하는 fallback 로직을 둘 수도 있습니다.

- **에러/도움말 메시지 개선 및 테스트 용이성 확보**  
  - `view-review.sh` 옵션 2에서 `git rev-parse` 실패 시에는 “유효하지 않은 커밋 해시”를 명시적으로 알려주고, 성공했는데 리뷰 파일이 없을 때만 “리뷰 없음” 메시지를 주면 UX가 좋아집니다.  
  - 두 스크립트 모두 완전 비대화형 모드(예: `--yes` 또는 `--non-interactive` 옵션)를 두면 CI나 자동 테스트에서 사용하기가 훨씬 편해집니다.
