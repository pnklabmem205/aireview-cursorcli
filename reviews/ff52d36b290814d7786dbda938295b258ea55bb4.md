## 리뷰 결과

**상태**: WARNING  
**요약**: 전반적으로 안전하고 목적에 맞게 동작하나, PR 본문 입력 로직과 일부 엣지 케이스에서 개선 여지가 있습니다.

**발견된 이슈**:
- **PR 본문 입력 로직 버그**:  
  `PR_BODY`를 입력받는 while 루프에서, 첫 줄이 빈 줄이면 즉시 break 되어 실제로는 PR 본문을 입력할 수 없습니다.
  ```bash
  while IFS= read -r line; do
      if [ -z "$line" ] && [ -z "$PR_BODY" ]; then
          break
      fi
      PR_BODY="${PR_BODY}${line}\n"
  done
  ```
  - 안내 메시지는 "Enter 두 번으로 종료"인데, 구현은 "첫 줄이 비어 있으면 바로 종료"라 UX/기능 모두 불일치입니다.
- **PR 본문 줄바꿈 처리**:  
  `PR_BODY`에 `\n`을 문자열로만 붙이고 있어, `gh pr create --body "$PR_BODY"` 호출 시 실제 줄바꿈이 아니라 `\n` 문자로 전달될 가능성이 큽니다. 다중 라인 본문을 의도한 동작과 맞지 않을 수 있습니다.
- **`set -e`와 `gh pr create` 에러 처리 충돌 가능성**:  
  `set -e`가 켜진 상태에서 `PR_URL=$(gh pr create ...)`가 실패하면, bash 버전에 따라 스크립트가 곧바로 종료되어, 뒤의 `if [ $? -eq 0 ]` 에러 처리 블록이 실행되지 않을 수 있습니다. 이 경우 사용자는 왜 실패했는지 메시지를 못 볼 수 있습니다.
- **GitHub URL 정규식 제한**:  
  `REMOTE_URL` 매칭 시 `.git` 접미사가 있는 형태만 처리하고 있어, `https://github.com/org/repo` 처럼 `.git`이 없는 일반적인 URL은 인식하지 못합니다. 이 경우에도 충분히 compare URL을 만들 수 있는데, 현재는 fallback 메시지로만 처리됩니다.
- **커밋 유무 체크 메시지 정확성**:  
  `COMMITS_AHEAD=$(git rev-list --count "$REMOTE/$DEFAULT_BRANCH"..HEAD ...)` 로 커밋 유무를 판단하는데,  
  - 원격 기본 브랜치(`$REMOTE/$DEFAULT_BRANCH`)가 아직 없거나, 리포 초기 상태 등 특수 상황에서는 실제로 커밋이 있어도 `0`으로 나와 “커밋할 변경사항이 없습니다”라고 잘못 안내할 수 있습니다. (크리티컬 버그는 아니지만 헷갈릴 수 있음)

**개선 제안**:
- **PR 본문 입력 로직 개선 (실제 “Enter 두 번으로 종료” 구현)**:
  ```bash
  echo -e "${YELLOW}PR 본문을 입력하세요 (Enter 두 번으로 종료):${NC}"
  PR_BODY=""
  EMPTY_COUNT=0
  while IFS= read -r line; do
      if [ -z "$line" ]; then
          EMPTY_COUNT=$((EMPTY_COUNT + 1))
          [ "$EMPTY_COUNT" -ge 2 ] && break
      else
          EMPTY_COUNT=0
      fi
      PR_BODY="${PR_BODY}${line}\n"
  done
  ```
  또는, 굳이 멀티라인이 필요 없다면 안내 문구를 “한 줄로 입력 (비우려면 그냥 Enter)” 정도로 바꿔 UX와 구현을 맞추는 것도 방법입니다.
- **줄바꿈 실제 적용**:  
  `\n` 문자열 대신 실제 줄바꿈을 사용하고 싶다면, `printf`를 활용하거나, `gh pr create`에 `--body-file`을 사용하는 방식(임시 파일에 본문 작성)으로 구현하면 줄바꿈이 정확히 반영됩니다.
- **`set -e` 하에서 에러 처리 안정화**:  
  `gh pr create` 호출 부분은 에러 처리를 명시적으로 하는 쪽이 좋습니다.
  ```bash
  set -e

  # ...

  echo -e "${YELLOW}PR을 생성하는 중...${NC}"
  if ! PR_URL=$(gh pr create --title "$PR_TITLE" --body "$PR_BODY" --base "$DEFAULT_BRANCH" 2>&1); then
      echo -e "${RED}❌ PR 생성 실패${NC}"
      echo "$PR_URL"
      exit 1
  fi
  echo -e "${GREEN}✅ PR이 생성되었습니다!${NC}"
  echo -e "${GREEN}$PR_URL${NC}"
  ```
- **GitHub URL 정규식 확장**:  
  `.git` 여부와 관계없이 동작하도록 패턴을 완화하면 좋습니다.
  ```bash
  if [[ "$REMOTE_URL" =~ github\.com[:/]([^/]+)/([^/.]+)(\.git)?$ ]]; then
      REPO_OWNER="${BASH_REMATCH[1]}"
      REPO_NAME="${BASH_REMATCH[2]}"
      PR_URL="https://github.com/$REPO_OWNER/$REPO_NAME/compare/$DEFAULT_BRANCH...$CURRENT_BRANCH"
      echo -e "   ${BLUE}$PR_URL${NC}"
  else
      # 기존 fallback 유지
  fi
  ```
- **커밋 유무 체크 로직 보완**:  
  “커밋할 변경사항이 없습니다”라는 메시지는 사실 “원격 기본 브랜치 기준으로 새 커밋이 없습니다”에 가까우므로,
  - 메시지를 더 정확하게 바꾸거나
  - 추가로 `git log "$DEFAULT_BRANCH"..HEAD` / `git status --porcelain` 등을 참고해 진짜 “푸시할 새 커밋이 있는지”를 판별하는 보완 로직을 넣는 것을 추천합니다.
- **테스트 용이성**:  
  - PR 생성 스크립트는 `DRY_RUN` 옵션(예: `DRY_RUN=1 ./scripts/create-pr.sh`)을 두고, 실제 `git push` / `gh pr create` 대신 수행할 명령만 출력하는 모드를 추가하면 로컬/CI에서 안전하게 테스트하기 좋습니다.
